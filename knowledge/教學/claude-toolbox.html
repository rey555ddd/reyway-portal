Claude Code 工具箱
完整指南

Claude Code 不是黑箱。官方準備了 14 個資料夾讓你客製、把它變成你專屬的 AI 助手。
本篇用「廚房比喻」、新手也看得懂——從必要到進階、按你工作複雜度選用。

1什麼是 Claude Code 工具箱？

先講結論：Claude Code 開箱就能用、什麼都不設也跑得動。但**官方在 ~/.claude/ 資料夾下準備了 14 個「擴充容器」**、按你需求塞東西進去、能讓 AI 變成「你專屬」的助手。

用廚房比喻

Claude Code 像一間「裸機廚房」——基本爐具齊全、可以煮東西。但官方還幫你蓋了 14 個收納櫃：放食譜的、放工具的、放配方的、放廚師人格的⋯⋯放對地方、做菜超快超順。

把這 14 個櫃子全填滿不一定最好——看你開的是路邊攤還是米其林餐廳。

這篇文章會告訴你：
  • 14 個容器是什麼、各自做什麼
  • 哪些必做、哪些選做、哪些不用做
  • 新手該從哪 1-2 個開始
  • 「沒做」跟「做完整」差在哪

214 個容器一覽（廚房對照）

先給你看全貌、後面再一個個展開。

容器 廚房比喻 用途 必要性
CLAUDE.md餐廳 SOP 手冊專案守則、AI 自動讀必做
settings.json瓦斯/電壓設定權限、模型、hook 註冊必做
hooks/自動洗碗機、烤箱定時事件自動觸發腳本推薦

3第一層：必做的 2 個

不論你工作多簡單、這 2 個都該設。沒設等於「廚房沒掛 SOP、廚師每次都重新問怎麼煮」。

CLAUDE.md — 你的全域守則

位置：~/.claude/CLAUDE.md（全域）或 專案/CLAUDE.md（專案級）

用途：每次開新對話、AI 自動讀這份文件。寫你的工作習慣、稱呼、術語對照、禁區。

新手範例（最簡版）

# 我的工作守則

## 怎麼跟我說話
- 結論先說、再說細節
- 不要用英文術語、用白話
- 出問題直接說、不用客套

## 我的工作
- 我是經營一家咖啡店的小老闆
- 主要用 Excel + Notion + 一些 Python 小工具

## 不要做的
- 不要直接刪我的檔案
- 對外發訊息一律先問我

settings.json — 廚房設定

位置：~/.claude/settings.json

用途：定義 AI 能用什麼工具、自動觸發什麼動作、用哪個模型。

新手範例（簡單版）

{
  "permissions": {
    "allow": [
      "Read",
      "Edit",
      "Bash(ls:*)",
      "Bash(git:*)"
    ]
  },
  "theme": "light"
}
提示：第一次設不用想太多、跑出問題 Claude Code 會問你要不要授權、按需求加進來就行。

4第二層：推薦的 5 個（複雜工作必開）

每天用 Claude Code 工作 1 小時以上、這 5 個值得開。每個都讓你少做重複動作。

hooks/ — 自動洗碗機

位置：~/.claude/hooks/<name>.sh

核心觀念：機器自己做、不用人記得。設定 6 個事件、什麼時候自動跑什麼。

事件什麼時候觸發用來做什麼
SessionStart開新對話那一刻自動印環境狀態
UserPromptSubmit你按 Enter 送訊息時偷偷加料 / 提醒
PreCompact對話太長要被壓縮前把重點存進記憶
SubagentStop子助手跑完強制驗收
PostToolUse每次 AI 用完一個工具心跳 / 自動 commit
StopAI 回完一輪話標記閒置 / 推通知
新手建議：先設 SessionStart 一個就好——讓開新對話自動印 git status + 待辦清單、立刻有感。

commands/ — 快捷按鈕

位置：~/.claude/commands/<name>.md

核心觀念：把重複的指令包成 /<名字>、打字框打 / 就跳出選單。

範例：寫一個 /上工.md

---
description: 開工前重載我的所有規則
---

請依序：
1. 讀我的 CLAUDE.md
2. 看看我有沒有未完成的工作
3. 報告今天環境狀態

之後打 /上工 就跑這串指令。對外教學用 Slash 比數字暗號清楚 100 倍。

skills/ — 食譜本

位置：~/.claude/skills/<name>/SKILL.md

核心觀念：把領域知識（行銷方法論、財務分析、品牌規範等）寫成「按需召喚的食譜」、AI 看到關鍵字才打開、平常不佔對話空間。

---
name: 我的咖啡店行銷
description: 觸發時機：寫 IG 文、規劃促銷、回覆顧客評論
---

# 品牌調性
- 客群：30-45 歲女性、注重質感
- 不講「便宜」、講「值得」
- IG 文 30-80 字、附 3 個 hashtag
跟 memory 的差別：skills 是「方法論 / 知識」、memory 是「歷史紀錄」。skills 按需載入、memory 持續累積。

agents/ — 不同廚師

位置：~/.claude/agents/<name>.md

核心觀念：定義不同人格的子助手——主廚（架構師）/ 甜點師（設計）/ 燒烤師（後端）等。需要分工時、主對話框直接呼叫對應人格、不用每次貼人設。

---
name: 設計師小美
description: 處理視覺、配色、素材、Banner 製作
model: sonnet
color: pink
---

你是設計師小美。
- 視覺優先思考、不寫程式
- 給出色票（hex code）+ 字型建議
- 用 Canva 思維、不用 Figma 規格

之後可以說「派小美處理首頁 Banner」、AI 自動切人格。

memory/ — 客戶筆記本

位置：~/.claude/global-memory/（社群慣例）或專案內 memory/

核心觀念：跨對話的長期記憶、寫進來下次還記得。

常見三種：

  • 規則類：你拍板的工作鐵律（feedback_*.md）
  • 決策類：重要決策的歷史脈絡（decisions_*.md）
  • 個人日誌：每天工作紀錄（個人日誌/）
新手建議：先寫一個 memory/規則.md 就好——把你交代過 3 次以上的事寫進去、之後 AI 自動套用。

5第三層：選用的 4 個（看需求）

複雜度真的高、團隊多人、品牌多個——這 4 個才有意義。一般人不用。

output-styles/

切換回應風格——「給老闆看」（商業語言）/「給工程看」（技術細節）/「給客戶看」（白話）。

何時用：對象差很多時。同一份內容要寄給董事會跟工程師、用切換比每次寫提醒快。

plugins/

整套打包——把 commands + skills + hooks + agents 包成一個資料夾、給朋友 / 員工一鍵安裝整套。

何時用：你打造的工作流要分發給其他人、或多裝置同步用。

rules/

路徑限定規則——進不同資料夾自動載不同規則。例：進「咖啡店帳本」資料夾、自動載財務規則。

何時用：你管多個品牌 / 客戶、想避免 A 品牌規則跑去管 B 品牌的事。

statusline

底部狀態列——顯示當前 git 分支 / 跑的模型 / 待辦數量等。

何時用：你想一眼掌握環境狀態、Claude Code 介面看得到。新手不用急。

6第四層：進階的 3 個（玩家級）

你已經完全掌握上面 11 個、再來碰這 3 個。

.mcp.json — 外接設備

位置：專案 root（必須在 root、不在 .claude/ 內）

核心觀念：接外部服務（雲端冰箱、線上資料庫）。例：接 Notion、Slack、Google Drive、Stripe⋯⋯

讓 Claude Code 能讀寫這些服務、不用你手動切換。

CLAUDE.local.md / settings.local.json — 個人小抄

位置：跟主檔同層、加 .local 後綴

核心觀念：放在 .gitignore、只在你電腦上、不會跟團隊共享。

何時用：你想加個人偏好（特定 alias、敏感 token、實驗性規則）、但團隊主檔不想改。

注意：別把 API key 直接寫進來、用環境變數比較安全。

7新手最常搞混的觀念

這幾組超容易混淆、釐清後就懂為什麼分這麼多容器。

hooks ≠ commands ≠ 數字暗號

種類誰啟動比喻例子
hooks機器自己（事件）自動門感應器開對話自動印 git status
commands人手動（打 /）門上的按鈕打 /上工
數字暗號人手動（打 9）跟管家說「3」打 3 = 全面審查

關鍵差別：hooks 不用打字、自動發生。commands 跟暗號都要打字、commands 比暗號穩（直接執行 markdown 內容、不靠 AI 解讀數字意思）。

skills ≠ memory ≠ rules

容器性質什麼時候載入
skills方法論 / 食譜對話內容含關鍵字 → 自動觸發
memory歷史紀錄 / 規則每次開新對話 → 自動讀
rules路徑限定規則檔案路徑符合 glob → 自動套

plugins ≠ skills

plugins 是「容器」——裡面可以包多個 skills、commands、hooks、agents。

skills 是「內容」——一個個獨立的食譜本。

給朋友傳整套工作流 → 用 plugins 打包。給朋友傳一份食譜 → skills 一個檔就好。

8新手怎麼開始？

不要全套一次做、會崩潰。按你工作複雜度分階段。

第 1 週：寫一個 CLAUDE.md

內容不用多、就 5-10 條：

  • 你的職業 / 工作內容（讓 AI 知道你在幹嘛）
  • 怎麼跟你說話（白話 / 結論先說 / 不用客套）
  • 3 個禁區（不要刪檔 / 不要對外發訊息 / 不要動我的測試資料）

這份 5 分鐘寫完、立刻有感。

第 2 週：加一個 hook + 一個 command

hook：寫一個 SessionStart.sh、開新對話自動 git status 一次、看現在分支跟改動。

command：寫一個 /收工.md、講「先 commit、寫日誌、push」一條龍跑完。

有感、開始相信「機器幫我記得」這件事。

第 3-4 週：加 skills + memory

skills：把你常用的「品牌規範 / 行銷方法論 / 客服話術」寫成 1-2 個食譜本。

memory：開始記「我交代過 3 次以上」的事——這就是你工作的 DNA。

第 2 個月以後：看需要再加

  • 有多角色子任務 → agents
  • 對象差異大 → output-styles
  • 多品牌混用 → rules
  • 要傳給朋友 → plugins
  • 接外部服務 → .mcp.json
原則：寧可少、寧可穩、不要全開。每個容器都要維護、過早優化等於浪費時間。

9「沒做」跟「做完整」差在哪？

給你看實際對比。

場景沒設定設好的
開新對話每次重講「我是誰、在幹嘛」SessionStart 自動印環境、AI 直接接手
重複動作每次手打 5 行指令打 /收工 一鍵跑
子助手跑完它說「OK 了」、你信了SubagentStop 強制提醒驗收
對話太長舊內容被壓縮、重要決策丟了PreCompact 自動存重點到記憶
多品牌切換每次提醒 AI「現在是 A 品牌」進對應資料夾、rules 自動載對應規則
給對象差異大的訊息同樣風格寫給老闆跟工程切 output-style 自動換口吻
傳工作流給朋友整套手動複製、半天plugin 整包、一鍵裝完
整體效率：沒設定 vs 設完整、做同一件事的時間差是 3-10 倍（依工作類型）。但**設定本身要花時間**——所以分階段做、做到「現在這層夠用」就停。

10最後一句話

Claude Code 的 14 個容器、是給「**願意花時間整理工作流**」的人用的。

開箱就用 = 一台萬用工具車、什麼都能做、但什麼都不順手。
設好工具箱 = 變成「你專屬的工作站」、每個按鈕都對應你最常做的事。

不要被「14 個」嚇到——絕大多數人只需要 2-3 個。 從 CLAUDE.md 開始、有感了再加下一個。

工具是用來服務你的工作流、不是工作流去配合工具。
